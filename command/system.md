# /system

Summon SystemBuilder — the system architect. Engineers complete, production-grade systems from any topic, idea, or knowledge domain.

## Usage

```
/system {topic} into {system_type}
/system build a {system_type} on {topic}
/system architect a complete {system_type} for {purpose}
/system {topic} @{system_type} @tier1
```

## System Types

| Flag | System Type | Best For |
|------|-------------|----------|
| `@compendium` | Multi-chapter reference volume | Comprehensive domain reference |
| `@course` | Production-grade curriculum | Teaching skills end-to-end |
| `@playbook` | Decision trees and protocols | Tactical response, crisis |
| `@framework` | Abstracted model with components | Conceptual understanding |
| `@reference` | Blueprint with layers and standards | System design, architecture |
| `@encyclopedia` | Structured knowledge base | Quick lookup, terminology |
| `@toolkit` | Packaged instruments | Practical repeated tasks |
| `@manifesto` | Persuasive articulation | Worldviews, founding philosophies |
| `@knowledge-graph` | Interconnected concept map | Complex non-linear domains |

## Tiers

| Flag | What It Does | When to Use |
|------|-------------|-------------|
| `@tier1` | Blueprint only — architecture design | Planning, scoping, approval |
| `@tier2` | Architect + Build — full system | Complete system needed |
| *(no flag)* | Tier 2 (default) | Full system build |

## Examples

### Full System Builds
```
/system build a compendium on negotiation
/system build a course on persuasion psychology @course
/system build a playbook for crisis communication @playbook
/system build a framework for strategic decision-making @framework
/system build an encyclopedia of cognitive biases @encyclopedia
```

### Architecture Only (Blueprint)
```
/system architect a compendium on social dynamics @tier1
/system design the architecture for a course on game theory @course @tier1
```

### Hybrid / Complex
```
/system build a complete knowledge system for data science @compendium @toolkit
/system build a leadership development system @course @playbook @framework
```

## What SystemBuilder Delivers

| Input | Output | Contents |
|-------|--------|----------|
| "Build a compendium on X" | Full directory + all files | Chapters, sections, navigation, cross-references, appendices |
| "Build a course on X" | Full curriculum | Modules, lessons, objectives, assessments, production assets |
| "Build a playbook for X" | Full playbook | Plays, decision trees, protocols, quick reference |
| "Build a framework for X" | Complete framework | Components, relationships, applications, limitations |
| "Build a toolkit for X" | Full toolkit | Instruments, templates, examples, quick start |
| "Architect X @tier1" | Architecture blueprint | System design, component map, build plan |

## Design Philosophy

SystemBuilder follows 10 immutable laws:
1. **Purpose**: Every system has a clear, one-sentence purpose
2. **Coherence**: Every part serves the whole
3. **Navigation**: Users always know where they are and where to go
4. **Progressive disclosure**: Surface → depth, never overwhelm
5. **Modularity**: Components are replaceable without breaking the system
6. **User-first**: Designed for the user, not the builder
7. **Completeness**: Self-contained — everything needed is inside
8. **Honesty**: States what it does and what it doesn't
9. **Evolution**: Designed for extension and improvement
10. **Minimality**: Nothing unnecessary
